Architecture Decision Record (ADR)
Title: Secure Password Storage (Hashing + Salting)
Date: February 16, 2026
Status: Accepted
Context
Passwords are sensitive credentials. Storing them in plain text or with weak hashing algorithms exposes users to severe risks if the database is compromised. Industry best practices dictate the use of strong, salted, and iterated hashing algorithms.
Decision
- Passwords will never be stored in plain text.
- We will use a hashing + salting strategy with a modern algorithm:
- PBKDF2 with SHA‑256 (100,000 iterations).
- Each password gets a unique random salt.
- Hash + salt stored in the database.
- Verification is done by recomputing the hash with the stored salt and comparing results.
- Future flexibility: The hashing algorithm can be swapped for bcrypt or Argon2 if required.
Consequences
- Security: Even if the database is leaked, attackers cannot easily recover passwords.
- Compliance: Meets modern security standards (OWASP, NIST).
- Maintainability: Clear separation of password hashing logic in a utility class (PasswordHasher).
- Performance: Iteration count balances security with acceptable login speed.
